# CS 410/510 — Data With Python: project policy

**Official syllabus:** [Course Syllabus](https://canvas.pdx.edu/courses/115534/assignments/syllabus)

This document summarizes expectations for **this repository** and aligns with the course syllabus (Spring 2026). If anything here conflicts with the syllabus or instructor directions, **follow the syllabus and instructor**.

## What is graded

The course centers on a **data project**. Typical weights (confirm on the syllabus):

| Component                                                                                                        | Weight (typical) |
| ---------------------------------------------------------------------------------------------------------------- | ---------------- |
| Project milestones (four phases: acquisition/wrangling; EDA/visualizations; deep inference; predictive modeling) | 40%              |
| Interactive model deployment (graduate section)                                                                  | 10% (grad)       |
| Final Jupyter notebook                                                                                           | 60%              |

The **primary graded artifact** is your **final Jupyter notebook** (and milestone submissions as specified). Files in this repo such as README, `docs/`, and Cursor rules exist to collaborate and to record policy—they are **not** substitutes for the notebook unless an assignment explicitly asks for them.

## Style and quality bar

- **Python style:** follow [PEP 8](https://peps.python.org/pep-0008/).
- **Substance:** clear narrative, reproducible steps, honest handling of data limitations, and proper attribution of sources.

## Honor code and attribution

- Submit **your own original work** that reflects **your** understanding.
- Cite external sources, datasets, documentation, and ideas you did not originate.
- Plagiarism or cheating can result in serious academic consequences per course policy.

## AI tools (including Cursor and chat assistants)

The syllabus distinguishes **how** AI may be used:

**Not acceptable for submission**

- Turning in assignments, **code**, or **written analysis** that was **directly generated** by an AI tool as if it were entirely your own unaided work.

**Permitted when documented**

- **Refinement:** improving clarity, grammar, formatting, or style of **your own** draft text or code.
- **Debugging:** helping find and fix errors when you are stuck, provided you understand the fix.

**Required documentation**

Any use of AI for **refinement** or **debugging** must appear in the submission under a **“Sources Used”** section, including:

1. The specific tool (e.g., ChatGPT, Gemini, GitHub Copilot, Cursor).
2. A brief description of how it was used.

### Sources Used — template

You can paste this into your notebook or written submission and fill it in:

```markdown
## Sources Used

- **Tool:** (e.g., Cursor, ChatGPT, Gemini)
- **How used:** (e.g., grammar and clarity edits on the introduction; suggested a fix for an off-by-one index after I pasted the traceback)
- **Scope:** (e.g., refinement only / debugging only — not full draft generation)
```

If you did **not** use AI for refinement or debugging on a given deliverable, state that explicitly or leave a short note such as “No AI tools used for refinement or debugging on this submission.”

## Suggested workflow for this repo

- Use **chat or the agent for questions**, explanations, and debugging hints.
- **Author notebook cells yourself**—type and structure the analysis you intend to submit; rework any suggestion until it is yours to explain.
- Keep **Sources Used** accurate and specific for every milestone and the final notebook.
